这次问题最开始看起来非常“玄学”：

- 有时连得上，有时连不上；
- 换几次环境、重启几次机器之后似乎又好了；
- 日志里没有特别明确的一条致命错误。

后来回头看，根因其实很朴素，主要集中在两处配置细节。

## 1. MongoDB 密码占位符写法错误

我当时在连接串里把 `<password>` 理解成“要保留尖括号的占位符”。  
正确做法是直接替换为真实密码，不要保留尖括号。

例如密码是 `123456`，应该写成 `123456`，而不是 `<123456>`。

为了避免把密码硬编码进源码，后面改成了命令行参数读取：

```js
const password = process.argv[2]
```

然后运行：

```bash
node filename.js yourpassword
```

这一步把“连接串格式错误”这类低级但高频的问题直接规避掉了。

## 2. 本地 DNS 配置不稳定

第二个问题是本机 DNS 解析层不稳定，导致外部依赖访问时好时坏。

在 Windows 下把 DNS 显式改到公共地址后，连通性明显稳定：

- `8.8.8.8`
- `8.8.4.4`

这一步虽然和业务代码无关，但对本地开发链路影响很大。

## 这次复盘最有价值的地方

当时最容易做错的一件事，是把问题当成“复杂系统故障”，不停换版本、重启和重复试错。

真正有效的是先把排查顺序固定下来：

1. 先看连接串和凭据格式  
2. 再看网络与 DNS  
3. 最后再考虑运行时和依赖版本

很多看起来“随机”的故障，最后都落在这几类基础配置上。
