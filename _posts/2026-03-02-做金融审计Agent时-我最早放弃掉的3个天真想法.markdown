很多 AI 项目在最开始的时候都会有一种很诱人的错觉：

只要把资料给够，把 prompt 写长一点，把模型换强一点，系统就会开始像一个真正的专业助手。

但做金融审计 Agent 之后，我很快放弃了几个特别典型、也特别天真的想法。

不是因为这些想法完全错，而是因为它们一旦进入真实场景，就会立刻碰到边界。

## 第一个天真想法：把 PDF 丢给模型，它自然会读懂

这是最早、也最常见的幻想。

看起来很顺理成章：财报就是 PDF，模型不是已经很强了吗，那直接上传文件，让它总结、找异常、出报告，不就好了。

但真正做起来就会发现，PDF 从来不是一个统一输入。

有些是数字版，有些是扫描版；有些表格规整，有些布局混乱；有些字段靠文本说明，有些字段散在不同页；有些关键关系不是单个数字，而是跨页勾稽。

这意味着，第一步根本不是“让模型理解”，而是“先把输入变成机器可以稳定消费的结构”。

所以 pdfplumber、OCR、结构化抽取，这些听起来不像 agent 核心的部分，反而是最早必须被认真对待的部分。

如果前面这一步不稳，后面的智能几乎都会变成幻觉。

## 第二个天真想法：把所有规则都给模型，它就会更全面

一开始很容易觉得，既然审计规则有 34 条，那最稳妥的方式当然是全部喂进去，让模型一次性通读、一次性评估、一次性出结论。

后来我很快发现，这其实是一个非常低效、也很容易失真的做法。

因为规则一旦太多：

- token 成本会迅速上升
- 模型注意力会变散
- 真正和当前企业相关的规则反而不突出
- 输出很容易开始泛泛而谈

这也是为什么 RAG 在这里不是点缀，而是必要结构。先检索最相关的少量规则，再让模型在局部上下文里评估，反而更接近真正可用的路径。

全面并不等于把所有信息同时塞进去。很多时候，真正的全面是“在正确时刻拿出正确部分”。

## 第三个天真想法：一次性给结论就够了

我最开始也很容易把“给出最终判断”当成终点。

但越往后越意识到，审计场景最关键的往往不是一句结论，而是过程里留下来的东西：

- 它为什么会这么判断
- 用了哪些规则
- 哪些地方证据不足
- 哪些地方需要二次验证
- 最终报告里哪些结论是高置信，哪些只是提示

如果系统只会给一个看起来很完整的结论，而没有过程留痕，那么它越像专业人士，风险反而越高。因为使用者最容易在这种“像”里放下警惕。

所以我后来越来越重视结构化输出、规则引用、过程记录，以及把“我不知道”也变成输出的一部分。

## 这些想法为什么会被放弃

因为真实场景会逼你承认一个事实：

审计不是问答，Agent 也不是一个会把所有复杂性自动吃掉的黑盒。

它更像是一条链路：

- 输入要先被整理
- 规则要被正确召回
- 评估要有步骤
- 输出要能追溯

一旦少了其中任何一段，系统都可能看起来能跑，但实际上不可靠。

## 做到后面，目标也会变

最开始做这类系统时，很容易把目标设成：“让模型像审计师一样回答。”

但做到后面，我觉得更准确的目标应该是：

让系统在专业流程里承担合适的那一部分工作。

比如：

- 帮助提取和整理
- 帮助召回相关规则
- 帮助完成初步评估
- 帮助生成格式化报告
- 帮助标出需要人工复核的风险点

这和“完全替代审计师”是两个完全不同的目标。

前者更慢、更克制，但也更真实。

## 所以现在回头看，最值得保留的是什么

如果要总结这几个被放弃的想法，我会说，它们共同的问题都在于过度相信模型会自动处理复杂性。

但真正有价值的系统，恰恰不是把复杂性藏起来，而是把复杂性拆成可以被机器和人共同处理的几段。

这也是我现在越来越相信的一件事：

做 Agent 最重要的，不是把 prompt 写得像魔法，而是把流程设计得足够诚实。
