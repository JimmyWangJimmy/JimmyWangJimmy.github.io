刚开始搭 Node.js 本地服务的时候，最消耗人的往往不是代码本身，而是那些看起来很碎的小问题。

比如服务改了为什么没重启，插件为什么忽然装不上，`npm init` 之后生成的默认脚本到底是什么意思，浏览器为什么没有跟着刷新。这些事情单看都不大，但特别容易反复打断节奏。

我后来会把这些问题当成一组典型的本地开发摩擦来看，而不是零散记忆点。

## 自动重启解决的是后端节奏，不是前端刷新

本地 Node 服务最先值得装的一个工具通常是 `nodemon`：

```bash
npm install --save-dev nodemon
```

它的意义很直接：后端文件改动后，不用手动停服务再重启。

但这里很容易产生一个误解，以为“自动重启”就等于“整个开发环境都会自动更新”。其实不是。`nodemon` 只负责后端进程重启，不会替你刷新浏览器页面。前端页面如果没有自己的热更新链路，还是要手动刷新。

这个区别很小，但不搞清楚就容易误判“为什么改了代码还是没变化”。

## ResponseMessage 是很细小但有用的调试入口

Express 响应里有一个不太常用但有时很方便的小点：

```javascript
response.statusMessage = 'Response You Like'
```

它可以覆盖默认的 HTTP Reason-Phrase，也就是状态码后面那段文本。

这不是业务必需项，但在本地调试或排查接口行为时，有时候能更快让你看清这一跳返回的到底是什么，而不是永远只看到一套默认语义。

## 很多插件安装失败，根因其实是代理残留

这类问题特别典型。

VS Code 插件突然装不上时，人很容易先怀疑：

- 扩展市场挂了
- 网络坏了
- 版本冲突

但有一种高频情况其实更简单：以前配过代理认证，后来环境变了，但 `setting.json` 里残留的 `http.proxyAuthorization` 还在。

这时最直接的检查方式就是打开 `setting.json`，找到这行配置，如果它已经不再有效，就把它清掉或设为 `null`。

很多“突然装不上”的问题，就是这种旧环境残留。

## npm init 默认脚本别无视

`npm init` 之后，很多人都会看到这个默认脚本：

```bash
"echo \"Error: no test specified\" && exit 1"
```

它不是报错，也不是 npm 坏了，而只是一个占位提醒：你还没有定义测试命令。

这件事虽然简单，但它其实对应一个很重要的习惯问题：项目一开始生成的默认结构，不要只是放着不管，而要尽快替换成你真正会用的脚本。

比如：

- `start`
- `dev`
- `test`
- `server`

这些脚本越早整理好，后面本地开发越不容易混乱。

## 这些小坑为什么总是反复出现

因为它们都属于一种非常典型的本地开发摩擦：

- 不足以构成大故障
- 却足够频繁地打断节奏
- 而且很多时候不会留下很明确的错误信息

所以它们最适合的处理方式，不是每次现场现查，而是尽早形成一套自己的默认排查顺序。

比如我现在遇到本地服务异常，会优先想：

1. 进程有没有真的重启
2. 浏览器是不是只是没刷新
3. 代理配置有没有残留
4. 脚本是不是还停留在初始化状态

这些问题一旦先想清楚，很多原本会消耗半小时的小故障，最后几分钟就能收掉。

###### 下载express
```bash
npm install express 
```

###### 引入日志记录中间件Morgan
```bash
$ npm install morgan
```

在`index.js`中引入
```javascript
const express = require('express')
const morgan = require('morgan') //引入日志记录中间件

const app = express()

app.use(express.json())
app.use(morgan('tiny'))//使用tiny格式记录
```

tiny格式：
```
method :url :status :res[content-length] - :response-time ms
```
日志输出示例:
```
GET /api/persons 200 223 - 5.611 ms
```

###### 解决跨域问题
前置知识：
same-origin policy:
- 一个源是指一个协议、主机和端口的组合
- 一个源的脚本只能访问同源的资源
> This means that a web application using those APIs can only request resources from the same origin the application was loaded from unless the response from other origins includes the right CORS headers.


eg: http://localhost:3001/index.html 与 http://localhost:3001/api/persons 是同源的
因为有1.同样的protocol(http) 2.同样的host(localhost) 3.同样的port(3001)


CORS: Cross-Origin Resource Sharing
[MDN-CORS](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS)
<img src="https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS/cors_principle.png">

在后端服务中引入cors
```bash
npm install cors
```

###### 使用Fly.io 部署
1. 安装flyctl
如果遇到无法识别pwsh,则需要安装最新powershell

2. 登录fly
```bash
fly auth login
```

3. 创建应用
```bash
fly init
```

注意在部署前进行以下修改：
1.在`fly.toml`中添加环境变量（与后端的端口一致）
```toml
[env]
PORT = "3000"
```
2.修改`index.js`文件中的端口号
```javascript
const PORT = process.env.PORT || 3000
```

4. 部署
```bash
fly deploy
```

5. 查看部署情况
```bash
flyctl ping -o personal
```

注意涉及到文件夹的部分最好在gitbash中操作。

可以使用script命令来简化操作
一个例子：package.json`中的scripts部分
