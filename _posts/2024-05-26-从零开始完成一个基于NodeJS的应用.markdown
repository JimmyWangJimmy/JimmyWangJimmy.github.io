“从零开始”最容易踩的坑是：  
一开始就想写完整业务，结果基础链路还没稳，后面每一步都在补前面的坑。

这篇记录的是我后来更认可的一条起步顺序：先把最小可运行链路打通。

## 1. 先把项目入口和脚本管理好

初始化项目：

```shell
npm init
```

然后在 `package.json` 里先把启动脚本写清楚：

```json
{
  "scripts": {
    "start": "node index.js"
  }
}
```

它的意义是把“怎么启动项目”从口头约定变成项目本身的一部分。

## 2. 开发期必须加热更新

开发阶段如果每改一次都手动重启，效率会很快掉下来。  
所以我会尽早加 `nodemon`：

```shell
npm install --save-dev nodemon
```

然后加一个开发脚本：

```json
{
  "scripts": {
    "dev": "nodemon index.js"
  }
}
```

这样后面调接口和改结构时，反馈会稳定很多。

## 3. 后端基础库别拖到最后

我通常会在起步阶段就把这三类依赖装上：

```shell
npm install express cors mongoose
```

原因很简单：

- `express` 负责最小 API 壳子
- `cors` 解决前后端分离时最常见的跨域问题
- `mongoose` 让数据库层尽早进入可验证状态

等到业务写一半再补这些，返工成本反而更高。

## 4. 先跑通一个最小 API，再扩展

最实用的方式是先验证两件事：

- `POST` 是否能写入
- `GET` 是否能读回

例如用 `.rest` 文件快速压测：

```http
POST http://localhost:3003/api/blogs HTTP/1.1
Content-Type: application/json

{
  "title": "Test",
  "author": "Test",
  "url": "Test",
  "likes": 0
}
```

```http
GET http://localhost:3003/api/blogs
```

这一步跑通，说明最核心链路已经闭环：

前端/请求端 -> 后端路由 -> 数据库 -> 返回响应

## 5. 这类项目早期最该避免什么

最该避免的是“业务写很多，但基础能力没收口”。  
常见表现是：

- 启动方式不统一
- 本地开发没有热更新
- 跨域临时 patch
- 数据库连接只在某台机器可用

这类问题不会立刻爆炸，但会在项目变大后拖慢每一次迭代。

## 这篇的核心结论

Node.js 项目从零开始时，先把链路打通比先写功能更重要。  
当 `scripts`、热更新、跨域、数据库和最小接口都稳定后，后面扩功能会轻松很多。
