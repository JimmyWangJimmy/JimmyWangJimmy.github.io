很多前后端联调卡住的时候，问题不一定在代码，而在于链路根本还没建立起来。

前端页面已经写了一半，组件状态也差不多了，但后端接口还没齐，或者数据结构还在变，这时候如果继续空等，前端开发会很快进入一种“页面看起来在做，实际上什么都联不起来”的状态。

JSON Server 对我来说最大的价值，就是在这种阶段里快速搭出一个最小可用的联调环境。

## 第一步：先把假的接口跑起来

在项目根目录创建 `db.json`，然后直接启动：

```shell
npx json-server --port 3001 --watch db.json
```

这样你马上就会得到一套可访问的本地接口，不需要先等真正的后端服务完成。

如果这个命令要频繁使用，更适合写进 `package.json`：

```json
{
  "scripts": {
    "server": "npx json-server -p3001 --watch db.json"
  }
}
```

这样以后直接 `npm run server` 就行。

## 第二步：让前端尽快开始发真实请求

一旦假接口起来，前端这边就别再继续用写死的数据渲染页面了，而是尽快把请求链路接上。

最常见的做法就是装上 `axios`：

```shell
npm install axios
```

然后把原本写在组件里的静态数组，逐步替换成真正的异步请求。

这一步表面上只是“先假装有后端”，但实际上它会提前暴露很多真实问题，比如：

- 请求地址怎么组织
- 响应结构该长什么样
- loading 和 error 状态有没有位置放
- 列表更新之后页面怎么同步

这些问题如果不在联调前期处理，后面后端真的接进来时，返工会更大。

## JSON Server 真正解决的是什么

我后来越来越觉得，JSON Server 最有价值的地方不是生成假数据，而是帮你尽快建立接口思维。

一旦接口先跑起来，很多原本模糊的问题就会被逼出来：

- 列表接口到底返回数组还是对象
- 分页信息放哪里
- 新增、删除、更新是否遵循同一套路径
- 前端状态到底是以接口为中心，还是仍然围绕本地变量堆起来

这些问题越早暴露，越容易在真正后端接入前统一掉。

## 它适合什么阶段，不适合什么阶段

JSON Server 特别适合：

- 前后端分工已经开始，但真实后端还没就绪
- 需要快速验证页面交互
- 想先跑通增删改查链路
- 想把请求、状态、组件之间的关系先理顺

但它不适合假装自己是真后端。它不能替代权限、校验、事务、复杂查询，也不该被拿来掩盖真实接口设计还没想清楚的问题。

更准确地说，它是一个过渡层，不是终点。

## 我现在怎么理解它

现在回头看，JSON Server 最值得用的地方，不在“省事”，而在于它能把“空等后端”变成“先搭出一条能走的路”。

在很多项目里，这种最小可用链路一旦先立起来，前后端的沟通成本会立刻下降很多。

因为大家终于不再只讨论“以后接口应该怎样”，而是可以围绕一个已经能跑起来的结构，继续修正和收敛。
