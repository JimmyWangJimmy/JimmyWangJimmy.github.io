前后端联调最怕的不是“慢”，而是“没有可验证链路”。  
JSON Server 适合在这个阶段快速建立一个过渡层。

## 1. 启动本地假接口

先准备 `db.json`，再运行：

```bash
npx json-server --port 3001 --watch db.json
```

建议写进脚本：

```json
{
  "scripts": {
    "server": "npx json-server -p3001 --watch db.json"
  }
}
```

## 2. 前端尽快切到异步请求

不要继续用写死数组渲染页面，直接接请求层。  
这样可以提前暴露真实问题：

- 路径命名是否稳定
- 返回结构是否一致
- loading/error 是否有状态位

## 3. 约定一份最小响应规范

即使是假接口，也建议先统一形状，例如：

```json
{
  "code": 0,
  "message": "ok",
  "data": []
}
```

这会降低后续接真后端时的替换成本。

## 快速排查清单

1. 端口是否冲突（3001 是否被占用）
2. 请求路径是否对应 `db.json` 顶层键
3. 前端 baseURL 是否写对
4. 列表为空时是否返回 `[]` 而不是 `null`

## 一句话结论

JSON Server 是联调“过渡层”，不是后端替代品。  
它最有价值的作用是把空等后端，变成可持续推进的前端开发。
