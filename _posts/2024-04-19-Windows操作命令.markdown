刚开始折腾 Windows 环境时，我总以为提效要靠更复杂的工具。后来发现，真正高频起作用的，反而是几个很小的命令行动作。

它们不炫，也不高级，但在排障、联调、搬文件、切目录这些具体场景里，能明显减少来回折腾。

## 1. 先把最基础的复制动作用顺

在命令行里，最常见也最容易被忽略的动作就是复制文件。

```shell
copy source target
```

例如：

```shell
copy E:\Source E:\Target
```

这个动作看起来很基础，但在下面这些时候很有用：

- 临时备份配置文件
- 快速替换某个测试版本
- 在不同目录之间做最小迁移
- 避免频繁手动拖拽导致路径搞混

很多时候，环境问题并不是出在技术复杂度，而是出在“我到底把哪个文件放到了哪里”。  
把这种最基础的文件操作放回命令行，路径关系反而更清楚。

## 2. 在目标文件夹里直接打开 CMD

这个动作特别适合排障。

在文件夹地址栏里直接输入：

```text
cmd
```

回车之后，打开的命令行当前路径就是这个目录本身。

这个习惯能省掉很多低价值动作，比如：

- 反复 `cd` 多层目录
- 因为路径切错导致命令执行位置不对
- 明明在查一个项目问题，却总在用户根目录里乱跑

很多命令执行失败，不是命令本身错了，而是人站错了目录。  
这类问题越早在入口处规避，后面越不容易乱。

## 3. 把文件夹直接拖进 VS Code，比想象中更有用

这件事我后来才意识到：  
其实可以直接把文件夹拖进 VS Code 里打开。

听起来像小事，但它的价值在于，很多排障场景本来就需要在“终端目录”和“编辑器视角”之间来回切。

如果打开项目这一步还在绕，很容易出现这种情况：

- 终端在一个目录
- 编辑器打开的是另一个目录
- 以为自己改的是当前项目，结果改错地方

直接拖进 VS Code，本质上是在缩短“我现在到底在操作哪个目录”这件事的确认路径。

## 为什么我后来会更重视这些小动作

因为真正做排障时，最怕的不是复杂，而是混乱。

很多人一遇到问题就急着搜错误信息、换版本、装新工具，但实际上最先该做的常常是：

- 确认当前目录
- 确认当前文件
- 确认复制和替换动作有没有做对

这些动作听起来不像“技术能力”，但它们直接决定你排障时会不会不断制造新的变量。

从这个角度看，这些命令行小动作的价值不在于“会不会用”，而在于它们能不能帮你把环境里的混乱降下来。
