原始记录只有一句“接口格式定义”和一个不完整示例。  
这在论文里能过关，但在前后端协作里几乎无法直接用。

这次把它整理成一份可联调的最小接口规范。

## 设计目标

- 前后端分离，统一 JSON 协议
- 分页接口结构固定，减少前端判断分支
- 错误码统一，便于日志检索和告警

## 接口约定

- Base URL：`/api`
- Content-Type：`application/json`
- 时间格式：`yyyy-MM-dd HH:mm:ss`
- 认证方式：`Authorization: Bearer <token>`

## 示例 1：分页查询学术活动

- Method：`POST`
- Path：`/api/academic-activities/table`

请求体：

```json
{
  "page": 1,
  "size": 10,
  "keyword": "机器学习",
  "fieldId": 3,
  "dateFrom": "2024-01-01",
  "dateTo": "2024-12-31"
}
```

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "total": 126,
    "records": [
      {
        "id": 1001,
        "title": "学术活动标题",
        "fieldName": "人工智能",
        "speaker": "张三",
        "startTime": "2024-05-01 14:00:00",
        "location": "A101"
      }
    ]
  }
}
```

## 示例 2：查询学术领域列表

- Method：`GET`
- Path：`/api/academic-fields`

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": [
    { "id": 1, "name": "计算机科学" },
    { "id": 2, "name": "数学" },
    { "id": 3, "name": "人工智能" }
  ]
}
```

## 错误码建议

- `0`：成功
- `40001`：参数校验失败
- `40101`：未登录或 token 无效
- `40301`：无权限
- `50001`：系统异常

## 联调检查清单

1. 路径是否统一挂在 `/api`
2. 分页接口是否统一返回 `total + records`
3. 空结果时是否返回空数组而不是 `null`
4. 错误返回是否保留 `code + message`

## 一句话结论

接口文档的价值不在“写了多少字”，而在“前后端能不能按文档一次联通”。  
只要路径、参数、响应和错误码四件事统一，联调成本会明显下降。
