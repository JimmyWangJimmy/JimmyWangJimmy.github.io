当时碰到的是一个很典型、也很烦人的问题：  
文章里中文本来是正常的，结果某次打开之后突然开始乱码。

这种问题的麻烦不在于它多复杂，而在于它特别容易让人误判。你会怀疑：

- 是 Jekyll 配置坏了？
- 是浏览器渲染问题？
- 是主题不支持中文？
- 还是文件本身已经被写坏了？

后来收敛下来，问题其实没有那么大，本质就是编码不一致。

## 当时的现象是什么

最直接的症状是：

- 英文正常
- 中文显示异常
- 文件内容看起来像没丢，但字符已经不对了

这类问题通常说明两件事里至少有一件不一致：

- 文件是怎么保存的
- 编辑器是怎么重新打开它的

## 这类问题为什么容易绕远

因为“乱码”看起来像前端显示问题，实际上经常是文件层问题。

如果不先确认编码，后面做的很多动作都会变成噪音，比如：

- 改主题
- 查 CSS
- 重启服务
- 重建站点

这些都可能完全碰不到真正的根因。

## 最后是怎么修掉的

我当时在 VS Code 里做的动作是：

1. 打开文件  
2. 点击右下角当前编码显示  
3. 选择 `Reopen with Encoding`  
4. 重新以 `UTF-8 with BOM` 打开并保存

修完之后，中文显示恢复正常。

这里真正值得记住的不是某个菜单项，而是排查顺序：

1. 先确认这是不是编码问题  
2. 再确认当前编辑器是以什么编码理解这个文件  
3. 最后才决定要不要重新保存成统一格式

## 这件事后来给我的提醒

很多环境问题的根因都不在“代码逻辑”，而在更底层的约定没有统一。

编码、换行符、路径分隔符、大小写、时区，这些都属于同一类问题：

- 平时感觉不到
- 一旦不一致就很折腾
- 真正有效的办法不是猜，而是先把基础约定确认清楚

所以现在再遇到类似问题，我通常不会先问“系统怎么坏了”，而是先问：

这个文件现在到底是按什么规则被读和被写的？

一旦这个问题问对了，很多看起来很玄的乱码问题就会快很多。
