很多人做数据项目，第一反应是往“更聪明”的方向冲。

比如更大的模型，更复杂的分类器，更好看的可视化，更像研究报告的结论页。  
这些当然重要，但我越来越确定一件事：**如果证据层没打牢，后面所有高级动作都只是浮在半空里。**

我最近回看 `THU-AgentWork` 这个仓库时，感受尤其强。

如果只看仓库名，它很像一个普通课程作业目录；  
但如果顺着 `README`、提交历史、清单文件和设计文档往下看，会发现它真正完成的事情并不普通。它已经把 **2025 年 A 股上市公司年报 PDF** 的抓取、去重、核查、目录治理和下一阶段解析契约，压成了一套可以继续长的底座。

这篇文章想讲的，就是这个过程。

不是“我做了一个很厉害的模型”，  
而是“我先把最脏、最容易被忽视、却最决定后续质量的那部分工程做好了”。

<h2 id="why-evidence-first">为什么我觉得数据项目第一步不是模型，而是证据层</h2>

做年报研究时，很多人天然会把注意力放在后半段：

1. 文本怎么做主题分析；
2. 风险披露能不能转成指标；
3. 管理层讨论与分析能不能拿来做情绪或景气判断；
4. 最后能不能写出像样的行业洞察。

这些问题都对，但它们默认了一件前提：**你手里的年报资产是干净的、完整的、口径一致的、出了问题时能回溯的。**

现实里，这个前提经常不成立。

一个很常见的误区是，把“下载了一批 PDF”当成“数据已经准备好了”。  
但两者之间其实隔着一整层工作：

1. 标题过滤是不是准确；
2. 摘要版有没有被剔除；
3. 同一公司同一年多个披露版本怎么处理；
4. 下载失败有没有被完整记录；
5. 后续解析结果能不能回到具体文件、具体页、具体版本。

这些问题听起来不性感，却决定了你后面做出来的任何统计和模型，到底是不是站在实物上。

我现在越来越相信，数据项目的顺序应该反过来：

先问证据层是不是可靠，  
再问分析层是不是聪明。

因为一个模型最怕的，不是能力不够，而是你根本不知道它到底在吃什么。

<h2 id="what-it-built">THU-AgentWork 到底已经做成了什么</h2>

从仓库现状看，`THU-AgentWork` 第一阶段已经完成得比“课程作业”这个标签听起来扎实得多。

根据仓库里的可核验证据，目前至少可以确认这些事实：

1. `README.md` 明确把范围限定为巨潮资讯上的 `2025 年年度报告` 正文 PDF，覆盖沪市、深市和北交所。
2. `reports_raw/manifests/download_manifest_2025_full_refresh.csv` 记录了 `1494` 份 canonical 年报。
3. `reports_raw/failures/download_failures_2025_full_refresh.csv` 当前记录数是 `0`。
4. `reports_raw/` 当前体量大约 `18G`，已经不是轻量样本。
5. `reports_raw/` 下现有 PDF 数量约 `5662`，说明仓库除了 canonical 版本，也保留了大量历史或替代版本以便回溯。
6. 当前 `git rev-list --count HEAD` 是 `97`，足以看出这不是一次性扔上来的结果，而是经过持续推进、修正和核查长出来的。

这几条放在一起，项目的性质就变了。

它已经不是“能抓一些文件”的脚本集合，  
而是一套更接近**证据层基础设施**的仓库：

1. 抓取链路明确；
2. 去重规则明确；
3. 清单和失败记录明确；
4. 目录治理明确；
5. 下一阶段解析契约已经文档化。

其中最关键的一点，是它对“canonical 版本”这件事的处理。

年报这类披露材料并不是天然稳定的。  
同一家公司、同一年，可能存在不同时间点披露的多个版本。如果没有“保留最新版本作为 canonical”的规则，你后面的研究结果就很容易出现两类问题：

1. 同一公司被重复计数；
2. 研究引用的文本和最终披露版本不一致。

所以仓库真正做对的一件事，不是“抓得多”，而是把“什么才算最终可研究样本”这件事先钉死了。

<h2 id="how-it-grew">这个仓库是怎么一步一步长出来的</h2>

我很喜欢从提交历史里看一个项目，因为真正的工程路径通常都写在里面。

`THU-AgentWork` 的成长轨迹非常真实，几乎能看出作者当时是怎么一边跑、一边卡住、一边修、一边继续往前推的。

### 1. 先搭最小可用抓取器

项目最早的重要节点，是主抓取脚本 `scripts/fetch_annual_reports_2025.py` 的出现。

这一步的目标非常克制，不是“直接做完研究”，而是先把最基本的链路打通：

1. 调用巨潮资讯历史公告接口拿候选公告；
2. 用 `2025年年度报告` 做标题筛选；
3. 排除摘要版；
4. 只保留 PDF；
5. 按公司和年份做去重；
6. 下载成功写 manifest，失败写 failure。

这里最让我在意的，不是脚本能跑，而是它从一开始就在输出**可核查结果**。  
manifest 和 failure 文件看起来只是 CSV，但本质上它们是在给这个项目补审计线索。

### 2. 发现“大任务一次跑完”不现实，于是开始分批推进

从后续一连串增量 batch 提交可以看出来，作者很快就接受了一个现实：  
这种量级的抓取任务，不适合等“全部完成后再整理”，而应该边跑、边落盘、边校验、边提交。

这其实是一种很成熟的工程习惯。

因为数据抓取不像写纯本地代码，它始终受到外部环境影响：

1. 网络抖动；
2. 源站响应不稳定；
3. 长任务中断；
4. 本地大文件堆积；
5. 推送阶段出新问题。

如果你还坚持“等全好了再说”，大概率只会把问题累积到最后一次性爆开。

### 3. 再往前一步，开始用 shard 并行

项目后面引入了按证券代码前缀分片抓取的能力，例如把 `000`、`002`、`300`、`600`、`601`、`688`、`920` 这些前缀拆成不同任务跑。

这一步的价值不只在于提速，更在于把风险拆开。

当抓取任务按 shard 独立运行时：

1. 单个前缀失败，不会拖垮全局；
2. 每个 shard 都能单独产出 manifest；
3. 断点续跑更容易定位；
4. 数据覆盖率能逐步往上堆，而不是等待一个理想中的“一次成功”。

这种做法背后的思维很值得借鉴：  
不是去想一个完美的大系统，而是把一个大问题切成一组可以独立收敛的小问题。

### 4. 仓库慢慢从“脚本目录”变成“资产仓库”

后面随着 `reports_raw/companies/`、`reports_raw/manifests/`、`reports_raw/failures/` 这些结构稳定下来，仓库的角色已经变了。

它不只是放脚本的地方，也不只是放下载结果的地方。  
它开始承担三层职责：

1. 原始资产保存；
2. 结果口径核查；
3. 下一阶段解析输入。

这也是我为什么说，它已经长成了一个“底座”，而不是一个“作业文件夹”。

<h2 id="where-it-bent">中间真正踩过哪些坑</h2>

如果只看最终结果，很容易误以为这是一条平滑路线。  
但提交历史和辅助脚本已经把弯路暴露得很清楚了。

### 1. 网络环境不是透明的

项目中途专门做过一类加固：显式关闭系统代理自动发现，避免 TLS 握手偶发卡死。

这看起来像一个很小的实现细节，但对抓取器来说，它往往比“业务逻辑对不对”还更先决定成败。

很多爬虫项目最后死掉，不是因为不会抓，而是因为作者假设运行环境是干净的、稳定的、可预测的。  
现实恰恰相反。只要任务足够长，环境摩擦早晚会出现。

所以这一步给我的启发很直接：  
**不要把环境问题当噪音，要把它当第一类工程对象。**

### 2. Git 对大批量 PDF 并不友好

仓库里出现 `scripts/push_untracked_pdf_batch.sh`，本身就说明项目踩到了大文件推送的现实问题。

这个问题特别典型：

1. 你本地已经抓到了很多 PDF；
2. 但一次性提交或推送，开始出现超时和摩擦；
3. 于是你不得不把“数据推送”本身也工程化。

换句话说，项目不仅在处理抓取问题，还在处理“如何把原始资产稳定地送进版本库”这个更脏的操作问题。

这是很多课程作业和 demo 项目里看不到的部分，但恰恰最有现场感。

### 3. 目录结构不是一开始就正确的

项目后面还专门做过目录重构，说明最早的文件组织方式并不完全适合规模化数据积累。

这件事我反而很认同。  
因为目录设计本来就很难在零数据状态下一次想对。通常只有当：

1. 文件数变多；
2. 版本变多；
3. 口径变多；
4. 下游开始真正消费这些文件；

你才知道什么样的结构扛得住。

所以这类重构不是返工，而是项目开始变真的标志。

### 4. 第一阶段做完，不等于第二阶段自然会发生

现在仓库里 `parsed_csv/` 仍然基本为空，只保留了 `.gitkeep`。这说明一个很关键的事实：

第一阶段“抓到并整理好原始资产”，和第二阶段“把 PDF 解析成结构化数据库”，中间隔着一整道新的工程门槛。

这个门槛不会因为你抓取做得很好就自动消失。

也正因为如此，后面出现的字段字典、schema、SQL 初稿和 parser implementation plan，才显得很重要。  
它们在做的，就是把“下一步要怎么跨过去”提前说清楚。

<h2 id="ai-collaboration">AI 在这个项目里到底扮演了什么角色</h2>

我最不喜欢的一种叙事，是把 AI 协作说成“AI 一把梭全做完了”。

看 `THU-AgentWork` 这个仓库，实际情况远不是这样。

如果把工作拆开，你会发现人和 AI 的分工非常清楚。

### 1. 人主导的是现场推进

抓取器什么时候先跑起来、什么时候该切 shard、什么时候该批量推送、什么时候该重构目录，这些都高度依赖实际运行反馈。

这种工作不是纯语言生成能替代的。  
它需要人对任务状态、运行环境、进度节奏和风险位置有持续判断。

所以这个项目里，真正把第一阶段推过去的，明显还是人的现场工程能力。

### 2. AI 放大的，是结构设计和下一阶段的组织能力

而在另一侧，仓库里的这些文档又很能说明问题：

1. `docs/superpowers/specs/2026-04-19-annual-report-data-design.md`
2. `docs/superpowers/specs/2026-04-19-annual-report-field-dictionary.md`
3. `docs/superpowers/specs/2026-04-19-annual-report-schema.md`
4. `docs/superpowers/specs/2026-04-19-annual-report-schema.sql`
5. `docs/superpowers/plans/2026-04-19-annual-report-parser-implementation.md`

这些材料的价值，不在于“替人写文档”，而在于它们把下一阶段要面对的问题组织成了清晰契约：

1. 要有哪些表；
2. 每张表该承载什么字段；
3. 解析链路如何分阶段；
4. 哪些测试应该先写；
5. 哪些输出必须能回到证据来源。

这恰好是 AI 在工程协作里最有价值的一种形态：  
**不是替你完成现场执行，而是帮你把未来执行的结构提前压清楚。**

### 3. 真正重要的是“把 AI 输出沉淀成仓库资产”

如果 AI 的输出只停留在聊天窗口，它再聪明，也只是临时讨论。  
但当这些输出变成：

1. 字段字典；
2. schema 文档；
3. SQL 初稿；
4. parser 计划；

它们才真正进入项目资产层。

这是我越来越看重的一点。  
与其说“AI 帮我写了点东西”，不如说“AI 参与产出了可版本化、可复查、可继续迭代的工程契约”。

这两者差别很大。

<h2 id="next-stage">这套底座接下来该往哪里长</h2>

从当前仓库状态看，最自然的下一步已经非常明确了：  
不是继续抓更多 PDF，而是把现有资产真正推进到结构化解析阶段。

我认为最值得优先做的，是下面四件事。

### 1. 先打通 parser 最小闭环

不要一上来追求全字段、全表完美落地。  
先把一条最小闭环跑通更重要：

1. PDF 发现；
2. 元数据归一化；
3. 页级文本抽取；
4. 章节切分；
5. 导出少量核心表。

只要这条线能稳定跑通，后面扩字段、补规则、加语义层就有了基础。

### 2. 把“证据回链”当成硬约束

任何结构化结果，最好都能回答这几个问题：

1. 它来自哪家公司、哪一年、哪一版披露；
2. 它来自 PDF 的哪一页；
3. 它对应的原文片段是什么；
4. 它是由哪个 parser 版本抽出来的。

这一步不是锦上添花，而是整个研究数据库能不能长期可信的关键。

### 3. 规则层和语义层要分开

我不建议把后续解析直接写成一锅粥。

更好的方式是：

1. 规则层负责章节锚定、页级清洗、数值和时间表达抽取；
2. 语义层负责主题归类、风险归因、跨公司叙事相似性比较。

两层分开，后面无论是 debug、重跑还是替换模型，代价都会低得多。

### 4. 尽快做一版“可研究输出”样本

哪怕只拿几十家公司、几个章节，也应该尽快产出一版真正能用于分析的表结构。  
因为只有当下游真的开始消费数据，你才会知道 schema 到底是不是对的。

这也是我对这种项目最现实的看法：  
**不要等到底座百分之百完美才让分析开始，而是要让底座和分析尽早相互校正。**

## 最后一句

如果让我用一句话概括 `THU-AgentWork` 现在最值得写的地方，我会说：

**它最有价值的，不是已经做完了什么炫目的分析，而是它把一条原本很容易散掉的数据抓取任务，压成了一套可以继续长、可以复核、可以通向下一阶段研究数据库的硬底座。**

这类工作通常不显眼，甚至看起来有点笨。  
但真正做过项目的人都知道，后面那些更聪明、更漂亮、更像成果展示的东西，往往都只能建立在这种“先把脏活做扎实”的基础上。
