很多人把 Obsidian 用着用着就卡住了。

最开始是兴奋期。看到一篇好文章、一个 GitHub 仓库、一串课程链接、几张截图，就想保存下来；文件夹越来越整齐，标签越来越多，插件越装越全，最后却发现一个很尴尬的事实：**资料是积累了，但判断没有积累，行动更没有积累。**

我最近重新整理自己的知识工作流时，刻意不再把 Obsidian 当成“一个好看的笔记软件”，而是把它当成一个 **LLM 可维护的知识操作系统**。这两种理解差别很大。

前者的重点是记录；后者的重点是路由、上下文和持续协作。  
前者关心“这条内容存在哪”；后者关心“这条内容现在该去哪里、以后谁来用、什么时候该重新出现”。

<h2 id="why-notes-fail">为什么大多数笔记系统最后都会失效</h2>

问题通常不在工具，而在默认工作假设错了。

多数人的默认假设是：  
“只要我把重要资料收集起来，未来总能用上。”

这句话听起来合理，实际问题很大。因为“收集”只是入口动作，不是知识动作。真正让系统失效的，通常是下面四件事：

1. 原始证据没有保留，后面回看时只剩二手摘要。
2. 概念和来源混在一起，越积越像资料仓库，越不像知识库。
3. 高价值资料和待办事项混在一起，最后所有东西都在争同一个注意力入口。
4. Agent 没有稳定上下文，每次都像第一次认识你一样重新理解需求。

我最近在自己的自主学习笔记里反复写到一个判断：**入口群不应该被当成待办清单，而应该被当成漏斗入口。**

这件事看起来很小，但它直接决定了后面的系统形态。

如果入口消息默认都进日程，你很快会被“我能替你学掉的东西”占满时间。  
如果入口消息先进入证据层，再由 Agent 判断是丢弃、沉淀、深挖还是排期，整个系统才开始像一个能协作的系统，而不是一个更复杂的收藏夹。

<h2 id="vault-architecture">我现在这套 Obsidian 知识库是怎么分层的</h2>

这套结构的启发来自 Karpathy 提到的 `LLM Knowledge Bases` 思路，但我实际落地时，做了更强的“行动路由”改造。

目前的核心分层是四块：

1. `raw/`：原始资料层  
2. `wiki/`：知识编译层  
3. `outputs/`：回答与产出层  
4. `_system/`：索引与健康检查层

这套结构的最简入口定义，写在仓库的 `README.md` 里。

### 1. `raw/` 不是草稿箱，而是证据层

我现在把网页文章、论文、仓库说明、图片、数据集、零散笔记都先丢进 `raw/`。这样做的理由很简单：**先保留原始证据，再允许 Agent 处理。**

这一步的价值不是整洁，而是防止后面出现最典型的知识污染：

- 只剩摘要，找不到原文
- 只剩观点，不知道是谁说的
- 只剩结论，不知道时间和上下文

所以 `raw/` 是知识库的输入层，不是长期表达层。  
它负责“保真”，不负责“优雅”。

### 2. `wiki/` 才是长期知识层

`wiki/` 下面我又拆成了三个子角色：

- `wiki/concepts/`：稳定概念
- `wiki/sources/`：来源笔记
- `wiki/maps/`：导航页和专题总览

这三个角色如果不拆开，后面一定会混。

概念页回答“这是什么”；  
来源页回答“这篇资料说了什么”；  
地图页回答“这几个主题怎么连起来”。

这一层的重点不是“写很多”，而是让 Agent 下次再来时，能快速判断：

- 哪些概念已经有共识表达
- 哪些资料只是单一来源
- 哪些问题还没有被串起来

### 3. `outputs/` 不应该混回知识库

这是我以前经常做错的一点。

回答、图表、幻灯片、临时分析，天然具有“任务上下文依赖”。  
如果把它们和概念页混在一起，知识库很快会失去长期稳定性。

所以我现在把它们都放到 `outputs/`：  
它们是**本轮任务的结果**，不是**长期知识的定义**。

### 4. `_system/` 是治理层，不是装饰层

这一层通常最容易被忽视，但从长期看它最重要。

如果没有索引、健康检查和基本治理脚本，一个知识库早晚会变成：

- 孤岛文件越来越多
- 重复概念越来越多
- 路径越来越长
- 链接越来越脏

而 LLM 在这种环境里工作，质量只会越来越差，因为上下文污染会越来越严重。

<h2 id="route-rule">真正关键的不是收集，而是路由</h2>

如果只说知识库结构，这篇文章还不够完整。  
我最近最确定的一个判断是：**资料系统的核心不是收集能力，而是路由能力。**

在我的工作流里，一条新资源进来后，必须先回答下面几件事：

1. 它是什么类型：文章、仓库、课程、截图、任务线索，还是纯噪音。
2. 来源是否可靠：有没有原文、有没有时间、有没有清晰出处。
3. 和当前目标有多近：是当前主线，还是远期冷库。
4. 我能不能替你学掉：如果可以，就不该默认占用你的时间。
5. 最终该进哪里：丢弃、沉淀、深挖、排期、汇报。

这套判断在我的笔记里已经开始变成更明确的“入口群信息操作系统”。  
对应的两份内部工作文档分别是 `2026-04-29-autonomous-learning-report.md` 和 `autonomous-agent-learning-plan.md`。

这里最重要的不是分类名称，而是一个更严格的原则：

> 能由 Agent 替你学习和消化的内容，不应该默认进入你的日程。

这条原则非常实用。

因为多数“知识焦虑”不是信息不够，而是把所有高价值链接都误判成“必须由我本人花完整时间处理”。  
结果就是：

- 日程被学习任务挤爆
- 真正需要你本人判断的事情反而被稀释
- Agent 变成了一个更积极地给你布置作业的人

这不是协作升级，这是注意力劫持。

所以我现在更愿意把 Agent 的职责定义成：

- 替你读
- 替你筛
- 替你建立知识链接
- 只把必须由你本人投入的部分升级为行动

<h2 id="memory-layers">Agent 为什么必须有分层记忆</h2>

很多人说 Context Engineering，本质上还是在说 prompt。  
我的判断更直接一点：**Context Engineering 的难点根本不是写 prompt，而是管理记忆。**

如果 Agent 没有稳定记忆，它每次面对同样的资料都会重复犯同样的错。

例如在我的实际场景里，Agent 必须长期记住这些约束：

- 工作日白天上班，晚上可能加班
- 早上起不来，不适合把学习安排在早晨
- 周中已有固定运动与课程安排
- 能由 Agent 自己学习的内容，不该默认排进人的日程
- 输出尽量中文、简单、可视化

这些东西不是“单轮对话上下文”，而是长期协作约束。  
如果不稳定保存下来，每次规划都会走回原点。

我现在更倾向于把 Agent 记忆至少分成五层：

1. 工作记忆：当前任务和最近上下文
2. 短期记忆：最近几天处理过的资料和待验证结论
3. 偏好记忆：用户节奏、表达偏好、时间边界
4. 长期记忆：长期主题、长期目标、重复问题
5. 经历记忆：哪些流程成功过，哪些坑已经踩过

没有这层分化，Agent 只能做到“会回答”；  
有了这层分化，Agent 才开始接近“会配合”。

<h2 id="daily-workflow">这套系统在日常里到底怎么跑</h2>

如果把它压缩成一条最小可用流程，大概是这样：

1. 新资源进来，先进入 `raw/`
2. Agent 保留来源、链接、时间、原文片段
3. Agent 生成或更新 `wiki/sources/` 来源页
4. Agent 更新相关 `wiki/concepts/` 概念页
5. Agent 把相关主题串到 `wiki/maps/`
6. 根据任务需要，把临时回答或图表放进 `outputs/`
7. 定期跑索引和健康检查，清理孤岛和重复

这个流程看起来不复杂，但它比“看一篇存一篇”多了两个非常关键的能力：

第一，**知识层和行动层分开了。**  
一篇资料很重要，不等于它今天就必须占用你的时间。

第二，**知识库开始具有再利用性。**  
下次再碰到类似问题时，Agent 不需要从零读起，而是可以基于已经沉淀的来源页、概念页和导航页继续推进。

这也是为什么我越来越不把知识库理解成“写给自己看的地方”，而是理解成“给未来的人和未来的 Agent 都能接手的地方”。

<h2 id="limits">这套方法的边界和下一步</h2>

这套方法并不神奇，它也有很清楚的边界。

第一，它很依赖输入纪律。  
如果入口层不保留原始证据，后面所有知识沉淀都会开始漂。

第二，它很依赖治理，而不是只依赖生成。  
LLM 可以很快帮你生成概念页，但如果没有 `_system/` 这一层的索引和健康检查，知识库会越来越像垃圾填埋场。

第三，它不能替代判断。  
Agent 可以替你读，替你筛，替你路由，但最终哪些主题值得长期投入，哪些结论应该进入公开表达，还是需要你本人拍板。

我接下来准备继续补三件事：

1. 把“入口资源五问法”写成更明确的判断表
2. 拿一条真实链接演示从入口消息到 Obsidian 来源页的完整流程
3. 把长期偏好和时间边界整理成稳定记忆清单

如果一句话总结这篇文章：

**我不是在把 Obsidian 变成一个更复杂的笔记本，而是在把它变成一个 Agent 能长期协作、能保留证据、能分清知识与行动的个人知识操作系统。**
