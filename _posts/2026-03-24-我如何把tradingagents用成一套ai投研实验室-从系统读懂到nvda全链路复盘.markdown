这篇文章不是对某个 GitHub 项目的泛泛介绍，而是一次很具体的使用复盘。

过去我对这类“多 Agent 投研框架”其实一直有一个保留：它们看起来很聪明，流程图也很复杂，但最后往往只给你一个像模像样的结论，却没有把“结论为什么会形成”讲清楚。真正让我愿意把时间投进去的，不是 `TradingAgents` 会输出 `BUY / SELL / HOLD`，而是它试图把一个基金公司内部本来就很分裂、很争吵、很依赖角色分工的判断过程，用一套可以检查、可以复盘、可以拆开的 Agent 工作流重新搭了一遍。

这次我没有只跑一个结果就停。我一共留下了四份文档，它们分别对应四个不同层级的理解：

1. `tradingagents_manual.md`：我先把系统本身读懂，确认它到底怎么编排 Agent、数据和状态。
2. `financial_terms_handbook.md`：我把报告里所有会阻碍阅读的金融术语单独拆出来，做成可回查手册。
3. `nvda_execution_report.md`：我把一次完整运行压缩成一份可快速浏览的执行复盘。
4. `nvda_detailed_execution_report.md`：我继续往下钻，把 Prompt、工具调用、数据格式、状态传播和最终决策都摊开。

所以你可以把这篇长文理解成那四份底稿之上的“总导读”。我不会删掉原始产出，也不会把它们揉碎成一个过于平滑的结论。相反，我更想保留这次使用里最有价值的东西：从系统层到术语层，再到执行层和 Prompt 层，最后拼成一个完整的使用经验。

<h2 id="why-tradingagents">1. 为什么我会认真用一次 TradingAgents</h2>

如果只是为了拿一个股票观点，其实完全没必要把系统搞得这么重。一个模型、一个财报摘要、几条新闻，再加一点技术指标，也能拼出一份“像那么回事”的报告。问题在于，这种单线程输出最大的问题不是不够长，而是不够立体。它很容易在一个方向上越写越像真理，却没有经过真正的对抗。

而市场里的很多判断，本来就不是“谁更聪明谁赢”，而是谁能更早发现另一套解释也说得通。技术面可能在恶化，基本面可能仍然很好；新闻面看起来是利多，资金面却未必买账；一个增长故事看起来很完整，但只要估值和库存开始一起变坏，结论就可能反过来。

这也是我这次为什么会认真对待 `TradingAgents`。它最吸引我的一点，是它没有把分析过程做成一个大 Prompt，而是把过程拆成了多个彼此有角色冲突的节点：四个分析师先各自说话，看多和看空研究员再互相辩，投研经理先给方向，交易员再把它转成可执行建议，最后还要经过激进、保守、中立三方风控再争一次，风控裁判才做最终裁决。

它不是在试图模拟“一个更强的大脑”，而是在试图模拟“一家会互相拉扯的研究团队”。这件事一旦做成了，哪怕最后结论还是错的，也会比一个平滑、单向、几乎没有反证的长报告更有用，因为你至少知道它是怎么错的。

<h2 id="institutional-debate">2. 这套系统真正模拟的，不是“分析股票”，而是“模拟机构争论”</h2>

`tradingagents_manual.md` 里最有价值的不是目录，而是它清楚地说明了这套系统想复刻什么。它不是“一个研究员写完，另一个人润色”，而是把传统基金公司的决策链拆成了 AI 版本：

- 先有四个分析师，分别看技术、情绪、新闻和基本面。
- 然后有看多和看空研究员基于同一批材料开打。
- 接着投研经理判谁的论证更有说服力。
- 再由交易员把方向变成真正的交易方案。
- 最后进入三方风控辩论，由风控裁判拍板。

这背后其实有一个很重要的思想转变：系统关心的不只是“谁能最先下结论”，而是“哪个结论能穿过更多层反驳还站得住”。这也是为什么它的价值不在最前面那几个分析师，而在后面几层对抗。

你可以把这套流程理解成一个漏斗：前面是分歧，后面才是收敛。真正值得看的是，每一层到底在淘汰什么。

![NVDA 决策漏斗](/assets/post-media/tradingagents-2026/nvda-decision-funnel.svg)

从系统实现上看，这件事也做得比较扎实。`manual` 里把几块关键设计交代得很清楚：

- 数据层不是写死的，走的是供应商抽象和路由模式，默认用 `yfinance`，技术指标通过 `stockstats` 批量计算。
- 编排层用的是 LangGraph，状态通过统一的 `AgentState` 传播，不是每个 Agent 各玩各的。
- 关键决策节点和一般分析节点用的是不同模型，形成了所谓的 `Quick Think / Deep Think` 分工。
- 记忆层没有搞得特别花，而是用 BM25 做离线检索，把过去经验喂回关键 Agent。

我比较认可这种分工。很多多 Agent 项目喜欢在 Agent 的数量上做文章，但真正让系统稳定的，往往不是“有多少个角色”，而是状态是否干净、数据是否可回放、分支是否可检查。`TradingAgents` 至少在这几个点上是认真设计过的。

<h2 id="four-documents">3. 四份文档分别解决了什么问题</h2>

如果只看最后一份 `nvda_detailed_execution_report.md`，你会看到极其完整的 Prompt、工具调用、CSV 返回值、指标结果、状态更新和 token 估算。它当然很全，但也很容易把人直接淹没。所以这次我保留四份文件，恰恰是因为它们各自承担了不同角色。

第一份是系统层。`tradingagents_manual.md` 回答的是：这套东西到底怎么搭起来，它想模拟谁，它的数据从哪来，状态怎么传，辩论何时停，记忆何时更新，Deep Think 和 Quick Think 为什么要分开。没有这份东西，后面所有执行报告都会像魔术。

第二份是认知层。`financial_terms_handbook.md` 其实是我在阅读报告时发现的一个很现实的问题：不是每个人都能自然读懂 `MACD`、`ERP`、`DOI`、`Put/Call Ratio`、`Forward P/E` 这些词，更不用说把它们和具体的交易判断连起来。于是我干脆把这些术语全部拆开，按技术指标、基本面、财务报表、期权、宏观和机构交易术语几类重做了一遍。这个动作本质上是在给多 Agent 报告做“阅读接口层”。没有这个接口层，报告越详细，普通读者越读不进去。

第三份是结果层。`nvda_execution_report.md` 的作用不是取代详细报告，而是先告诉你发生了什么。它把整个分析压缩成一条流水线：技术分析偏空，情绪分析偏 HOLD，新闻分析认为区间震荡，基本面却明确看多；然后 Bull/Bear 辩论里 Bear 获胜，Research Manager 给出 SELL，Trader 继续 SELL，三方风控也最终都站到了偏空一侧，最后 Risk Judge 输出 `SELL`。这是一份“先看剧情，再钻细节”的报告。

第四份才是机制层。`nvda_detailed_execution_report.md` 让我真正觉得这次复盘值得留下，因为它没有停留在“哪个 Agent 说了什么”，而是一路往下追到：Prompt 是怎么写的，第一次 `tool_call` 调了什么，股价 CSV 长什么样，技术指标如何被批量请求，消息何时清除，Bull 的上下文里到底塞了哪些报告，Risk Judge 最终读到了多长的辩论文本，Signal Processor 又是怎么把几千字决策压成一个单词 `SELL` 的。

这四份东西合在一起，才构成一次完整的使用经验。少掉任何一层，最后都会变形。只剩系统说明，会变成纸面设计；只剩术语手册，会变成金融入门；只剩执行报告，会变成“看个热闹”；只剩详细报告，则会变成工程细节洪水。

<h2 id="nvda-sell">4. NVDA 这次为什么最后会落到 SELL</h2>

真正让我觉得这套系统有意思的，不是它最后输出了 `SELL`，而是它并没有一开始就一边倒。恰恰相反，它前面几层是明显分裂的。

技术分析师给的是 `BEARISH`。原因很直接：价格跌破 200 日均线，50 日均线下行，10 EMA 疲弱，MACD 是近一年极端负值，RSI 也没回到 50 以上。这部分读起来是典型的技术面空头叙事。

情绪分析师却没有顺着技术面直接做空，而是给了 `HOLD`。这在我看来很重要，因为它捕捉到的是另一种现实：GTC、大模型、AI 基建这些叙事并没有真正消失，市场情绪是在分裂，而不是彻底崩塌。它不是说“看多”，而是说“这不是一边倒的恐慌场”。

新闻分析师更接近一种观望判断。他看到的不是单一利空，而是宏观环境、AI 基建、地缘风险缓解、板块轮动同时存在，最后把 NVDA 放在了一个区间震荡框架里。这种判断听起来不刺激，但很像真实研究里经常出现的那类结论：不是不知道方向，而是知道触发条件还没齐。

最反直觉的是基本面分析师，它明确给出了 `BUY`。而且它不是没理由：营收增长、毛利率、净利率、ROE、自由现金流、低负债，这些数字单拿出来都非常强。换句话说，系统前四个分析师里，至少有一个是在很认真地替 NVDA 的多头叙事辩护。

这时多 Agent 的价值才真正开始显现。因为如果系统停在这里，你得到的只会是“四份各说各话的报告”。真正有用的是后面的对抗。

Bull Researcher 的逻辑是典型成长股辩护：capex 已经签约，CUDA 护城河很深，财务壁垒很厚，技术破位只是颠簸，不是崩塌。Bear Researcher 则抓住了另一组更致命的信号：盈利预期在下修，超级云厂商更愿意投自研芯片，渠道库存风险上来，现货价格走弱，而这一次的技术破位也是十年里少见的三线空头排列。

Research Manager 最后站队 Bear，并不是因为基本面不强，而是因为它认为“强基本面”已经不足以抵消“边际恶化”。这点非常关键。真实市场里很多大票都不是死在绝对值差，而是死在方向开始转坏。只要市场开始怀疑未来的增长质量，再漂亮的当前利润也可能顶不住估值压缩。

Trader 后面又额外加了五个实时数据点，包括 Put/Call ratio 抬升、现货价格继续下跌、10 年期美债收益率回升、社交情绪衰减等。这一步我很喜欢，因为它说明 Trader 不是 Research Manager 的复读机，而是在把“研究判断”往“交易时机”上压。

最后进入三方风控辩论，系统没有简单粗暴地说“既然看空，就狠狠干”。激进派想大仓位做空，保守派强调仓位和对冲，中立派要求等跌破关键位再动。Risk Judge 最终虽然还是给出 `SELL`，但已经不是一个情绪化的 SELL，而是一个带条件、带仓位约束、带对冲结构、带分批计划的 SELL。

这才是这套系统这次最值得看的地方。它最终输出的不是一个态度，而是一套风险定义方式。

<h2 id="changed-my-mind">5. 这次使用里最让我改变看法的地方</h2>

第一点，是我对“多 Agent 是否只是形式主义”的看法有一点改变。

以前我会觉得，多 Agent 很容易沦为把同一件事说四遍，然后再找一个“裁判模型”做总结。但这次 NVDA 案例里，前序分析师的确产生了真实分歧，而且这种分歧不是噪音，而是金融决策里本来就存在的结构性冲突：技术面可以先坏，基本面可以暂时仍强，情绪和新闻又可能悬在中间。只要后面的辩论不是假辩论，这种架构是有价值的。

第二点，是我意识到“术语层”其实是这类项目非常容易被忽视的一层。

如果我不单独整理 `financial_terms_handbook.md`，整套输出其实很难对更广的人群真正可用。很多人不是看不懂模型，而是看不懂金融语言。你让一个系统吐出一份长报告很容易，但让一份长报告真正进入人的脑子里，中间还差一层翻译器。那层翻译器不是装饰，它决定这套东西最终是“能被引用”，还是“只能被惊叹”。

第三点，是我更确信这类系统的价值不在自动交易，而在可检查的研究过程。

我不会因为它给了一个 `SELL` 就相信它能替我做交易。真正有价值的是，当它做出这个 `SELL` 时，我能回头看到是哪几个节点让结论发生了偏转，是 Bear 论证赢了，还是 Trader 增加的实时数据改变了时机判断，还是风控把一个高仓位方案压成了低仓位分批策略。只要这个过程可见，它就不是黑箱。

第四点，是我也看到了它很现实的边界。

这次详细报告里估算的 token 消耗已经不低，整条链路并不轻；数据层主要依赖 `yfinance`，适合实验和复盘，不适合拿来做高置信度机构级执行；而且很多判断仍然建立在 Prompt 质量、新闻质量和指标解释的稳定性之上。换句话说，这套东西非常适合做“AI 投研实验室”，但不应该被误读成“自动赚钱机器”。

<h2 id="replication-guide">6. 如果你也想复现，我建议怎么读、怎么用、怎么避免误判</h2>

如果你是第一次接触这个项目，我不建议直接从最详细的那份报告开始。那样很容易被 Prompt、tool call 和长文本淹掉。更好的顺序其实是反过来的。

先看 `nvda_execution_report.md`，因为它能最快告诉你这次到底发生了什么。你先要知道系统最后为什么会从多空分裂收敛到 `SELL`，否则后面的细节没有依托。

再去看 `financial_terms_handbook.md`。这不是降级阅读，而是清理摩擦。如果术语层没打通，你很难真正判断 Bull 和 Bear 的论证质量。

然后去读 `tradingagents_manual.md`。到这一步，你已经知道结果长什么样，再回头看系统设计，就会更容易理解为什么它要用四个分析师、两层辩论、两类模型，以及为什么它要把状态拆得这么细。

最后再读 `nvda_detailed_execution_report.md`。这一步不是为了“看更多细节”，而是为了确认你是否真的信任这个系统的过程。如果你在这里发现某个 Prompt 设计得太强势，或者某个工具调用的时间窗选得不合理，那么你就已经开始真正使用它了，而不只是观看它。

如果是从实践角度出发，我对这类项目的建议很简单：

1. 先把它当成研究辅助，不要当成自动交易替身。
2. 先跑你熟悉的票，因为你需要有能力反驳它。
3. 先关注分歧是怎么形成的，不要只盯最后那个 `BUY / SELL / HOLD`。
4. 先把术语和数据层补齐，再决定是否继续加模型和角色。

最后我想说的是，这次让我真正留下印象的，不是 `TradingAgents` 多先进，而是它让我更明确了一件事：AI 在这类复杂场景里最值得追求的，不是更像一个“会直接给答案的人”，而是更像一个“能把过程摊开让你检查的系统”。

这恰好也是我愿意保留这四份原始文件、不把它们直接揉成一份短文章的原因。因为真正有价值的，不是最后那一个词，而是你怎么一步步走到那个词。
