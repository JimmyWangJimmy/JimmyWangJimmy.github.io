# 博客内容地图 2026-03

这份内容地图用于把博客从“时间倒序的个人技术笔记”收束成 3 条长期主线：

- 拆问题
- 与AI共生
- 阅读与思考

目标不是机械分类，而是建立一个稳定、可持续写下去的个人叙事：

- `拆问题`：我如何把现实里不工作的东西修好
- `与AI共生`：我如何和 AI 一起工作、开发、学习
- `阅读与思考`：我如何输入、整理、形成判断

## 1. 三个栏目怎么定义

### 拆问题

这个栏目不只写 bug。

它覆盖的是真实世界里一切“坏掉、卡住、半通半断、配置错位、环境脆弱”的问题，包括：

- 环境变量
- 代理与网络
- IDE / MCP / OpenClaw 工具链
- 部署链路
- API / 权限 / schema / 配置问题
- 项目开发中的调试、定位、修复

这个栏目的判断标准不是“有没有报错”，而是：

> 这篇文章是不是在把一个复杂问题拆开、定位、收敛，并最终修好。

建议语气：

- 少“教程感”，多“现场感”
- 少泛泛总结，多根因与决策

### 与AI共生

这个栏目不写泛 AI 新闻，也不做空泛观点堆砌。

它更具体地讨论：

- 我如何和 AI 协作开发
- AI 工具链怎么搭
- Agent / MCP / OpenClaw 怎么真正接入工作流
- AI 项目如何落地，而不是停在 demo
- AI 改变了什么，没改变什么

这个栏目真正要写的是“人机协作方式”，而不是“AI 很强”。

建议语气：

- 写工具，也写方法
- 写产品，也写使用中的摩擦
- 写效率，也写边界

### 阅读与思考

这个栏目比“读书”更宽。

它用于承接：

- 读书笔记
- 观影随想
- 音乐、播客、输入摘录
- 月度复盘
- 方法论和判断变化

这个栏目不要求高频，但必须稳定。它决定博客是不是只有“手”和“工具”，还是也有“脑”。

建议语气：

- 少摘抄，重在自己的判断
- 可以短，但要有密度

## 2. 现有文章初步归类

### 拆问题

核心代表：

- `彻底终结 MCP 连接地狱：Windows 环境下的路径排查与环境优化实战`
- `OpenClaw Discord 排障实录：Bot 在线但不回复的根因与修复`
- `解决ERROR: Permission to .git denied to deploy key fatal: Could not read from remote repository.`
- `一次debug经历`
- `Day 31 in March 2024`（编码问题修复，可重写后收录）
- `建站学习`（如果后续补全为“为什么文章显示不全”可纳入）

可继续整理为更成熟排障文的旧稿：

- `Windows操作命令`
- `Node.js WebServer`
- `JSON-Server`
- `Mongoose Module Set`
- `Vue2.6引入Vue-Echarts`

这类旧文的问题是：

- 记录感强，成稿感弱
- 结论有，但根因和上下文不够完整

后续策略：

- 不急着删
- 选有价值的重写成“问题背景 -> 排查过程 -> 根因 -> 修法 -> 复用建议”

### 与AI共生

现有可直接归入：

- `Spring AI Demo`
- `OpenClaw Discord 排障实录：Bot 在线但不回复的根因与修复`
- `彻底终结 MCP 连接地狱：Windows 环境下的路径排查与环境优化实战`

适合由 GitHub 项目延展成新系列的来源：

- `OpenClaw-KnowHowHub`
- `EagleEyeLite`
- `EagleEye`
- `PrometheusTrap`

这条线最重要的不是“技术点”，而是“你怎么和 AI 一起做事”。

建议未来优先把这条线写成下面几种文章：

- 工具接入实录
- 工作流设计
- 人机协作边界
- Agent 项目拆解

### 阅读与思考

现有可直接归入：

- `2025年——又一春.`
- `坠落的审判`

适合作为种子素材的公开豆瓣内容：

- 书：`杨振宁传`、`GPT图解`、`宏观经济学通识课`、`因子投资：方法与实践`
- 影：`探求自我的世纪`、`双相青春`
- 音：`Namgar - Nomad`、`Little Dragon Tales: Chinese Children’s Songs`、`一意孤行`、`Earth Breath`、`客途秋恨`、`诚实假日酒店`、`覚醒`、`Murder Most Foul`、`哪吒`

这个栏目不建议做成“读后感仓库”，而更适合：

- 月度输入
- 最近在看 / 在听 / 在想
- 一本书或一部电影如何改变了当下的判断

## 3. GitHub 素材如何转成文章

### OpenClaw-KnowHowHub

适合栏目：`与AI共生`

可写方向：

- 为什么 OpenClaw 不应只是一个聊天入口，而应该是“人和 agent 共用的知识操作系统”
- 把公共知识入口做成可供 agent 消费，到底改变了什么
- 知识库写给人看，和写给 agent 看，有哪些结构差异

### EagleEyeLite

适合栏目：`与AI共生`

可写方向：

- 做一个金融审计 agent，为什么不能只靠“问答”
- 从 OCR / PDF / RAG 到结构化输出，一条链路里最脆弱的环节是什么
- 在合规 / 审计场景里，AI 真正有价值的是哪里，最容易失真的是哪里

### EagleEye

适合栏目：`与AI共生`

可写方向：

- 为什么复杂金融文本不能只靠一次性 prompt
- “增量底稿”和“留痕”为什么是 AI 审计系统的核心，而不是附加项
- 针对城投债等长文本，如何设计跨页勾稽和规则注入

### financial-penalty-api

适合栏目：`拆问题`

可写方向：

- 从采集、清洗到 API 服务，监管处罚信息系统是怎么搭起来的
- 多源异构数据里，最难的不是爬，而是标准化和可信度
- 一个数据型后端项目，为什么调度、清洗、索引和服务层要一开始就想清楚

### PrometheusTrap

适合栏目：`与AI共生`

可写方向：

- 如果自然语言成为“攻击面”，Prompt-to-Earn 到底在防什么
- 为什么说 LLM 逻辑层不是传统智能合约的简单替代
- AI + Web3 项目里，哪些地方是叙事，哪些地方是工程

### 个人 GitHub 账号本身

适合栏目：`阅读与思考` 或 `与AI共生`

可写方向：

- 从课程练手、全栈笔记到 agent 项目，这几年我的技术兴趣是怎么迁移的
- 我为什么开始从“学技术”转向“搭工作流”

## 4. 豆瓣素材如何搬到博客

原则：

- 搬“自己的判断”，不搬平台的模板文案
- 不机械同步，而是二次整理
- 用豆瓣做素材库，不把博客做成豆瓣镜像

适合的内容形态：

- `月度输入：这个月我在读、在看、在听什么`
- `一本书 / 一部电影 / 一张专辑带来的一个判断`
- `输入摘录：最近在反复想的 3 个片段`

建议优先写的 4 个方向：

- `GPT图解` -> 不是再讲模型原理，而是补一篇“我如何把模型理解转成工作流理解”
- `杨振宁传` -> 从个体轨迹切入“长期主义、学科训练、选择”
- `探求自我的世纪` 或 `双相青春` -> 做成轻量观影札记
- 音乐清单 -> 做成一篇“最近循环的歌，和对应的状态”

## 6. 首页和栏目页应该怎么落

首页不再强调“技术博客”本身，而强调 3 条线：

- `拆问题`：从故障、代理、链路、配置到根因与修复
- `与AI共生`：工具、工作流、Agent、协作方式
- `阅读与思考`：书、影、音，以及方法论沉淀

首页文案建议：

> 记录我如何拆开复杂问题、和 AI 一起工作，以及如何形成自己的判断。

栏目页的使用方式：

- `拆问题`：优先放最强排障文和项目修复文
- `与AI共生`：放 AI 工具链、Agent 项目、协作方法
- `阅读与思考`：放输入、摘录、复盘、判断

## 7. 现在不建议做的事

- 不要再继续用 `jekyll update` 这种默认分类
- 不要给所有旧文章强行硬归类后就结束
- 不要把豆瓣内容整页镜像到博客
- 不要把 `与AI共生` 写成模型资讯搬运

## 8. 直接执行建议

最合理的顺序是：

1. 先把现有高价值文章映射到 3 个栏目
2. 首页内容分区改名为 `拆问题 / 与AI共生 / 阅读与思考`
3. 做 3 个独立栏目页
4. 先写完上面列出的 8 篇里的前 3 篇，再继续批量整理旧稿

当前判断：

- `拆问题` 会是博客最强的一栏
- `与AI共生` 会是博客最有时代感和辨识度的一栏
- `阅读与思考` 会决定这个站是不是只有技术信息，而没有人的轮廓
